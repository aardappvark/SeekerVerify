Incident Response Plan

Seeker Verify
Version 1.0.0 | February 2026

1. Scope and Objectives

1.1 Scope

This Incident Response Plan (IRP) covers all security, privacy, and operational incidents related to the Seeker Verify Android application, its published source code, its third-party dependencies, and its presence on the Google Play Store. It applies to the sole developer and publisher, MidMightBit Games.

1.2 Objectives

  • Detect, assess, and respond to security and privacy incidents in a timely manner
  • Minimise impact on users and their data
  • Meet regulatory notification requirements (OAIC, GDPR supervisory authorities, US state attorneys general)
  • Preserve evidence for investigation and regulatory compliance
  • Restore normal operations as quickly as possible
  • Learn from incidents to prevent recurrence

Email: aardappvark@proton.me

Role: All incident response functions (detection, analysis, containment, eradication, recovery, notification)

2. Incident Classification

Incidents are classified by severity to determine response urgency and notification requirements.

Priority Severity Description Response Time Examples
P1 Critical Active exploitation affecting user security or data integrity. Requires immediate action. Within 1 hour of detection Malicious code in published App build; compromised signing key; active data exfiltration vulnerability
P2 High Vulnerability or compromise that could affect users but is not yet actively exploited. Requires urgent action. Within 4 hours of detection Critical vulnerability in a dependency; compromised RPC endpoint serving malicious data; supply chain compromise detected before release
P3 Medium Issue that degrades security posture or user trust but does not pose an immediate threat. Within 24 hours of detection Third-party API exposing user IPs in an unexpected way; minor vulnerability in a dependency; policy violation discovered
P4 Low Minor issue with limited security or privacy impact. Addressed in normal development cycle. Within 7 days Informational security advisory for a dependency; Play Store policy update requiring minor changes; RPC endpoint deprecation notice

3. Response Procedures

3.1 Data Breach Response

Trigger: Discovery that user data (wallet addresses, cached SGT data) has been or may have been accessed by an unauthorised party, through a vulnerability in the App, a compromised build, or other means.

Phase 1: Detection and Assessment (0-1 hours)

  1. Confirm the breach: Verify the report, reproduction steps, and scope
  2. Assess the data affected: Determine which data elements are exposed (wallet addresses, SGT cache, engagement data)
  3. Assess the number of users potentially affected
  4. Determine whether the vulnerability is in the App code, a dependency, or the Android platform
  5. Document all findings with timestamps

Phase 2: Containment (1-4 hours)

  1. If the vulnerability is in the App: Prepare a hotfix build immediately
  2. If the vulnerability is in a dependency: Assess whether the dependency can be updated or removed
  3. If a published build is compromised: Request emergency removal from Google Play Store
  4. If a signing key is compromised: Revoke and rotate the key (requires new app listing)

Phase 3: Eradication and Recovery (4-48 hours)

  1. Develop, test, and publish a patched version of the App
  2. If applicable, issue a staged rollout to monitor for regressions
  3. Verify the fix resolves the vulnerability
  4. Update the App's changelog to describe the security fix (without disclosing exploit details)

Phase 4: Notification (within 72 hours)

  1. Notify regulatory authorities as required (see Section 4)
  2. Notify affected users via Play Store update notes and GitHub advisory
  3. Publish a security advisory on the GitHub repository

Contextual Note: Because Seeker Verify stores only wallet public addresses (which are already publicly visible on the Solana blockchain) and has no server infrastructure, the practical impact of a data breach is inherently limited. The wallet address cannot be used to access funds or control assets. However, all breaches will be treated seriously and reported in accordance with regulatory requirements.

3.2 RPC Endpoint Compromise

Trigger: A third-party RPC endpoint (Solana public RPC, Helius) or price API (CoinGecko, Jupiter) is compromised, returning manipulated data, logging excessive user data, or exhibiting malicious behaviour.

Phase 1: Detection and Assessment (0-2 hours)

  1. Verify the compromise report: Cross-reference data from the suspected endpoint against known-good sources (e.g., Solscan, another RPC provider)
  2. Determine the type of compromise: Data manipulation, excessive logging, service injection, DNS hijack
  3. Assess user impact: Could users have made decisions based on incorrect data?
  4. Determine how many users may be affected (based on RPC provider selection)

Phase 2: Containment (2-8 hours)

  1. If one RPC provider is compromised: Push an update that defaults to the alternative provider
  2. If a price API is compromised: Disable the affected price feed and display a warning to users
  3. Notify users via the App (if push update possible) or via GitHub/Play Store notes
  4. Contact the affected service provider to report the issue

Phase 3: Recovery (8-72 hours)

  1. Monitor the affected endpoint for resolution
  2. Re-enable the endpoint only after confirming the compromise is resolved
  3. If the provider is unreliable, evaluate alternative providers for future use
  4. Update the Privacy Policy if the third-party service list changes

3.3 Malicious Dependency

Trigger: A dependency used by the App is found to contain malicious code, a critical vulnerability, or has been compromised via a supply chain attack.

Phase 1: Detection and Assessment (0-2 hours)

  1. Identify the affected dependency and version
  2. Determine whether the affected version is included in the published App build
  3. Assess the potential impact: data exfiltration, code execution, privilege escalation
  4. Check security advisories (GitHub Security Advisories, NVD, OSV) for details
  5. Determine whether the App's usage of the dependency triggers the vulnerability

Phase 2: Containment (2-12 hours)

  1. If a patched version of the dependency exists: Update immediately and publish a new build
  2. If no patch is available: Assess whether the dependency can be temporarily removed or replaced
  3. If the vulnerability is critical and actively exploited: Request emergency App update via Play Store
  4. Pin a known-safe version of the dependency in the build configuration

Phase 3: Recovery (12-72 hours)

  1. Publish the patched App build to the Play Store
  2. Conduct a review of all other dependencies for similar issues
  3. Update dependency management practices if a process gap is identified
  4. Publish a GitHub security advisory if relevant

3.4 App Store Takedown

Trigger: Google removes or suspends the Seeker Verify listing from the Play Store due to a policy violation, regulatory action, or other reason.

Phase 1: Assessment (0-4 hours)

  1. Review the Google Play Console notification to understand the specific policy violation or reason
  2. Determine whether the takedown is related to a security issue, content policy, or regulatory request
  3. Assess whether existing users are affected (existing installs typically continue to work)
  4. Document the takedown reason with screenshots and timestamps

Phase 2: Response (4-48 hours)

  1. If a policy violation: Make the required changes to bring the App into compliance
  2. If a regulatory request: Consult the relevant regulation and determine obligations
  3. If a false positive: Prepare and submit an appeal through the Google Play Console
  4. Submit the corrected build or appeal for review

Phase 3: Communication (24-72 hours)

  1. Notify users via the GitHub repository if the takedown is extended
  2. If applicable, provide an alternative distribution channel (direct APK from GitHub Releases)
  3. Update governance documents if the takedown reveals a compliance gap

4. Notification Requirements

Critical: Regulatory notification timelines are strict. The 72-hour clock begins at the time of becoming aware of a notifiable breach, not at the time of completing the investigation.

Jurisdiction Authority Timeframe Threshold Method
Australia Office of the Australian Information Commissioner (OAIC) Within 72 hours (as soon as practicable after becoming aware) Eligible data breach likely to result in serious harm (Notifiable Data Breaches scheme, Part IIIC of the Privacy Act 1988) OAIC Notifiable Data Breach form at oaic.gov.au
EU/EEA Relevant GDPR Supervisory Authority Within 72 hours of becoming aware (GDPR Article 33) Personal data breach unless unlikely to result in a risk to the rights and freedoms of natural persons Notification to the supervisory authority of the lead establishment, or the authority of the affected data subject's member state
United Kingdom Information Commissioner's Office (ICO) Within 72 hours of becoming aware (UK GDPR Article 33) Personal data breach unless unlikely to result in a risk to the rights and freedoms of natural persons ICO breach reporting tool at ico.org.uk
United States State Attorneys General (varies by state) Varies by state (typically 30-60 days; some states require faster notification) Breach of personal information as defined by each state's breach notification law. Note: wallet public addresses alone may not meet the threshold in most states, as they are not typically classified as personal information under US state laws. Written notification to the AG of each state where affected residents reside. Some states have online portals.
Brazil Autoridade Nacional de Protecao de Dados (ANPD) Within a reasonable timeframe (LGPD Article 48) Security incident that may result in relevant risk or damage to data subjects Notification to ANPD and affected data subjects

4.1 User Notification

In addition to regulatory notifications, affected users must be notified when:

  • The breach is likely to result in a high risk to their rights and freedoms (GDPR Article 34)
  • Required by the OAIC under the Australian Notifiable Data Breaches scheme
  • Required by applicable US state breach notification laws

User notification channels:

  • Primary: Play Store app update description and in-app notification in the next release
  • Secondary: GitHub Security Advisory on the repository
  • Tertiary: Direct communication via community channels if available

6. Post-Incident Review

6.1 Review Process

A post-incident review must be conducted within 14 days of resolving any P1 or P2 incident, and within 30 days for P3 incidents. The review must cover:

  1. Timeline reconstruction: Detailed chronology of the incident from detection to resolution
  2. Root cause analysis: What caused the incident? Was it a code defect, dependency issue, configuration error, or external factor?
  3. Detection assessment: How was the incident detected? Could it have been detected earlier?
  4. Response assessment: Was the response timely and effective? Were there any delays or gaps?
  5. Impact assessment: What was the actual impact on users, data, and the App's reputation?
  6. Lessons learned: What can be improved in the App, its dependencies, or the incident response process?
  7. Action items: Specific, time-bound actions to prevent recurrence

6.2 Documentation

All incidents (P1-P4) must be documented with:

  • Incident identifier and classification
  • Date and time of detection
  • Date and time of resolution
  • Description of the incident and its root cause
  • Actions taken during response
  • Notifications sent (regulatory, user, advisory)
  • Post-incident review findings and action items

6.3 Governance Updates

After any P1 or P2 incident, review and update the following documents as needed:

  • Risk Register — Update risk assessments, add new risks if identified
  • Privacy Policy — Update if processing activities or third-party services change
  • Data Processing Register — Update if data flows or processing activities change
  • Legitimate Interest Assessment — Review if the incident changes the balancing test considerations
  • This Incident Response Plan — Update procedures based on lessons learned
