Legitimate Interest Assessment

Seeker Verify
Version 1.0.0 | February 2026

This Legitimate Interest Assessment (LIA) documents the analysis conducted by MidMightBit Games to determine whether legitimate interest under GDPR Article 6(1)(f) is the appropriate legal basis for processing wallet public addresses within the Seeker Verify application.

1. Purpose Test

1.1 What Is the Processing?

Seeker Verify processes a single data element: the user's Solana wallet public address. This address is used to:

  • Query the Solana blockchain to verify ownership of a Seeker Genesis Token (SGT)
  • Retrieve and display on-chain data including SKR token balance, staking position, rewards, and SOL balance
  • Resolve .skr domain names associated with the wallet
  • Calculate community fleet position and adopter tier based on publicly available on-chain data
  • Generate an on-device Season 2 airdrop prediction based on on-chain activity
  • Fetch USD price equivalents for SOL and SKR from third-party price APIs

1.2 What Is the Legitimate Interest?

The legitimate interest is providing a companion app service that allows Seeker device owners to verify their SGT status and view their on-chain portfolio data in a convenient, unified interface. This serves both:

  • The controller's interest: Providing the core functionality of the App as described to users
  • The user's interest: Easily accessing and understanding their own publicly available on-chain data without needing to navigate raw blockchain explorers

1.3 Is the Processing Necessary for That Purpose?

Yes. The wallet public address is the sole identifier on the Solana blockchain. Without it, the App cannot query any on-chain data. There is no alternative identifier that could achieve the same purpose with less data. The wallet public address is the minimum data element required.

2. Necessity Test

2.1 Is This the Least Intrusive Way to Achieve the Purpose?

Yes, for the following reasons:

  • Minimal data collection: Only the wallet public address is collected. No name, email, device ID, location, or other personal information is requested or stored.
  • Local-only storage: The wallet address is stored exclusively on the user's device using AES-256 encrypted SharedPreferences backed by the Android Keystore. No data is transmitted to any MidMightBit Games server.
  • No server infrastructure: The App has zero server-side components. All processing occurs on-device.
  • No analytics or tracking: No analytics SDKs, advertising SDKs, crash reporting, or telemetry of any kind are embedded in the App.
  • Read-only operation: The App never creates blockchain transactions or modifies any on-chain state.

2.2 Could Consent Be Used Instead?

While consent (Article 6(1)(a)) could theoretically be used, legitimate interest is more appropriate because:

  • The user voluntarily initiates the wallet connection through an explicit action (tapping "Connect Wallet" and approving in their wallet app)
  • The processing is minimal, expected, and directly requested by the user
  • The data (wallet public address) is already publicly visible on the Solana blockchain
  • The power dynamic is balanced — the App is free, open source, and the user can disconnect at any time
  • Relying on consent for processing that is inherently expected by the user would create unnecessary friction without improving data protection

3. Balancing Test

3.1 Nature of the Data

  • The wallet public address is a pseudonymous identifier. It does not directly identify a natural person by name.
  • The address is already publicly visible on the Solana blockchain to anyone with an internet connection.
  • No special category data (Article 9) is processed.
  • No criminal offence data (Article 10) is processed.

3.2 Reasonable Expectations of the Data Subject

  • Users download a blockchain companion app and voluntarily connect their wallet. They reasonably expect the App to use their wallet address to query on-chain data.
  • The App's description on the Google Play Store clearly states its purpose: SGT verification and portfolio display.
  • The wallet connection flow requires explicit user action: tapping "Connect Wallet" and then approving the connection in their wallet app (e.g., Phantom, Solflare).
  • The processing aligns precisely with what the user expects when performing this action.

3.3 Impact on Data Subjects

  • Positive impact: Users gain convenient access to their on-chain data, SGT verification, and community position.
  • Negative impact: Minimal. The wallet address is already public. Local encrypted storage reduces risk compared to server-side storage. No profiling or automated decision-making occurs beyond what the user explicitly requests.
  • Risk of harm: Low. Even in a worst-case device compromise, the exposed data (a public wallet address) is already available on public blockchain explorers.

3.4 Relationship Between Controller and Data Subject

  • Direct relationship: The user installs the App and actively connects their wallet.
  • No power imbalance: The App is free, has no premium tiers, and does not gatekeep any functionality behind data collection.
  • Full user control: The user can disconnect their wallet and delete all data at any time with a single tap.

3.5 Additional Factors

  • No data is shared with third parties beyond what is inherent in standard internet protocol (IP address exposure during RPC calls).
  • No data aggregation or cross-referencing with other data sources occurs.
  • No data is used for marketing, profiling, or any secondary purpose.
  • The App is open source, allowing full transparency into how data is handled.

4. Safeguards

The following safeguards are in place to protect data subjects' rights and interests:

4.1 Technical Safeguards

  • AES-256 encryption: Wallet address stored using Android EncryptedSharedPreferences with AES-256-GCM (values) and AES-256-SIV (keys)
  • Android Keystore: Encryption keys are hardware-backed where supported
  • HTTPS/TLS: All network communications are encrypted in transit
  • No cleartext traffic: The App enforces HTTPS for all connections
  • No server-side storage: Zero server infrastructure eliminates server-side breach risk

4.2 Organisational Safeguards

  • Privacy Policy: Comprehensive privacy policy accessible within the App and on the website
  • Transparent processing: All data visible to the user within the App at all times
  • Easy opt-out: One-tap wallet disconnect immediately deletes all stored data
  • Data minimisation: Only the wallet public address is collected; all other displayed data is fetched on-demand from the blockchain
  • Open source: Full source code available for inspection

4.3 Rights Facilitation

  • Right to erasure: Disconnect Wallet or Delete All Data in Settings
  • Right to access: All data visible in-app (Identity, Portfolio, Settings screens)
  • Right to object: Disconnect wallet at any time
  • Right to restriction: Disconnect wallet to cease all processing
  • Right to portability: Wallet address is already in the user's possession
  • Contact: aardappvark@proton.me for any data protection requests

5. Conclusion

Outcome: Legitimate interest is the appropriate legal basis.

The processing of wallet public addresses under GDPR Article 6(1)(f) is justified because:

  • The purpose is clear, specific, and directly serves the user's interest
  • The wallet public address is the minimum data necessary and is already publicly available on the blockchain
  • Users voluntarily initiate the processing and can cease it instantly
  • The impact on data subjects is minimal to positive
  • Robust technical and organisational safeguards are in place
  • The processing aligns precisely with the user's reasonable expectations
  • No data is transmitted to MidMightBit Games or any third-party for secondary purposes

The legitimate interests of the controller and the data subjects are not overridden by the data subjects' rights, freedoms, or interests. This assessment will be reviewed annually or when significant changes are made to the App's data processing activities.
